## 1. Purpose

This standard defines the policies and procedures for managing artifacts generated by <G@FT.ai> Studio's
Continuous Integration and Continuous Delivery (CI/CD) pipelines. Effective artifact management is
essential for:

- **Traceability:** Linking artifacts back to specific code commits, builds, and requirements.
- **Reproducibility:** Ensuring builds can be reproduced reliably.
- **Dependency Management:** Managing versions of shared libraries and services.
- **Release Management:** Providing stable and versioned artifacts for deployment and distribution.
- **Storage Efficiency:** Managing the lifecycle of artifacts to control storage costs.

## 2. Core Principles

- **Versioning:** All published artifacts **MUST** be versioned using **Semantic Versioning (SemVer) v2.0.
  0**. Versioning **MUST** be automated based on **Conventional Commits** (see `../tooling/
tool-001-conventional-commits-standard.md`).
- **Immutability:** Published artifacts **MUST** be immutable. Once an artifact version is published, it
  **MUST NOT** be overwritten or modified. If changes are needed, a new version must be published.
- **Centralized Storage:** Artifacts **SHOULD** be stored in a centralized, secure, and version-aware
  artifact repository.
- **Naming Conventions:** Artifacts **MUST** follow consistent and descriptive naming conventions.
- **Access Control:** Access to artifact repositories **MUST** be controlled to ensure only authorized
  processes or users can publish or consume artifacts.
- **Lifecycle Management:** Define retention policies for artifacts to manage storage consumption.

## 3. Standard Artifact Repository

- **Primary Recommendation:** **GitHub Packages** and **GitHub Container Registry (GHCR)** are the
  **standard preferred artifact repositories** for <G@FT.ai> Studio.
  - **Rationale:**
    - Tight integration with GitHub repositories, Actions, and Teams.
    - Supports various package types (npm, NuGet, Maven, RubyGems, Docker containers).
    - Simplified permission management via GitHub Teams.
    - Likely cost-effective, especially considering included storage/transfer with GitHub plans (to be monitored).
    - Aligns with our principle of leveraging the GitHub ecosystem where possible.
- **Alternatives:** Other self-hosted (e.g., Sonatype Nexus, JFrog Artifactory - OSS versions if available
  and suitable) or cloud-specific artifact repositories (e.g., AWS ECR/CodeArtifact, GCP Artifact Registry,
  Azure Artifacts) **MAY** be considered _only if_ GitHub Packages/GHCR present significant limitations for
  specific artifact types, scale, or advanced features not available. Any deviation requires an ADR and
  approval from DevOps Strategy (Gem D) and Architecture (Gem I).
- **Large Game Build Artifacts/Assets:** For very large game build artifacts (e.g., multi-GB client builds
  for different platforms) or large binary art assets not suitable for Git LFS, a dedicated binary storage
  solution (e.g., AWS S3 with versioning, or a specialized game asset management system) might be necessary.
  This will be defined in a separate standard if required. For now, GitHub Releases can be used for
  distributing final game builds.

## 4. Versioning Strategy

- All artifacts intended for consumption by other components, deployment, or release **MUST** be versioned
  using **Semantic Versioning (SemVer) v2.0.0** (`MAJOR.MINOR.PATCH`, e.g., `1.2.3`).
- Version bumping (`PATCH`, `MINOR`, `MAJOR`) **MUST** be automated in CI/CD pipelines based on the
  analysis of commit messages adhering to the **Conventional Commits Standard** (`../tooling/
tool-001-conventional-commits-standard.md`).
  - `fix:` type commits trigger a `PATCH` bump.
  - `feat:` type commits trigger a `MINOR` bump.
  - Commits with `BREAKING CHANGE:` in the footer (or `type!:`) trigger a `MAJOR` bump.
- **Pre-release Versions:** For development or testing builds, pre-release versions (e.g., `1.0.0-alpha.
1`, `1.0.0-beta.commitSHA`) **MAY** be used. These should clearly indicate their non-production status and
  ideally include commit SHA or build number for traceability.
- **Build Metadata:** Build metadata (e.g., `+build.123`, `+git.commitSHA.short`) **MAY** be appended to
  SemVer versions but does not affect version precedence.
- **Tooling:** Tools like `semantic-release` (for Node.js/Python/etc.) or equivalent language-specific
  tools integrated into CI/CD pipelines **MUST** be used to automate version calculation and tagging.

## 5. Naming Conventions for Artifacts

Consistent naming is crucial for discoverability and automation.

### 5.1. General Format (Guideline)

`<organization_scope>/<repository_or_project_name>-<component_name>:<version>` (for packages/images)
`<repository_or_project_name>-<component_name>-<version>-<platform>-<arch>.<extension>` (for executables/archives)

### 5.2. GitHub Packages / GHCR

- **Packages (npm, NuGet, Maven, etc.):**
  - Scope: `@GenCr-ft` (or your GitHub organization name).
  - Name: Follows the repository name or a clear component name, e.g., `@GenCr-ft/
gcp-aethel-client-shared-utils`, `@GenCr-ft/gencraft-api-contracts-ts`.
  - Version: SemVer (`1.2.3`).
- **Container Images (Docker):**
  - Registry: `ghcr.io/GenCr-ft/` (or your GitHub organization name).
  - Image Name: `repository_name` or `service_name`, e.g., `ghcr.io/GenCr-ft/gencraft-service-auth`.
  - Tag: SemVer (`1.2.3`), `latest` (for the most recent stable release on `main`), branch names (for
    development builds, e.g., `feat-new-login-flow`). **Production deployments MUST use immutable SemVer tags.**

### 5.3. API Contracts (`gencraft-api-contracts`)

- Published artifacts (e.g., OpenAPI specs, generated SDKs) should be clearly named and versioned.
- **Example SDK Package (npm):** `@GenCr-ft/gencraft-api-contracts-clientside-ts@1.0.0`
- **Example Contract File (GitHub Release/Package):** `gencraft-api-openapi-v1.2.3.json`

### 5.4. Game Builds (Client/Server Executables)

- Distributed via GitHub Releases or a dedicated distribution platform.
- **Naming:** `GenCr@ft-Client-vX.Y.Z-<platform>-<architecture>.<zip|exe|dmg>`
  - e.g., `GenCr@ft-Client-v0.1.0-win-x64.zip`
  - e.g., `GenCr@ft-Server-v0.1.0-linux-x64.tar.gz`

## 6. Artifact Lifecycle and Retention

- **Production Artifacts:** Artifacts deployed to production or released publicly **MUST** be retained
  indefinitely or for a very long, defined period.
- **Development/PR Artifacts:**
  - Artifacts generated from feature branches or Pull Requests (e.g., pre-release packages, test builds)
    **SHOULD** have a shorter retention period to manage storage costs.
  - GitHub Packages and GHCR offer some control over this, or specific cleanup actions/scripts might be needed.
  - **Recommendation:** Define a retention policy (e.g., 30-90 days for PR-specific artifacts) and
    implement automated cleanup where possible.
- **S3 Bucket Versioning (if used for state/artifacts):** Enable versioning on S3 buckets used for
  critical artifacts or Terraform state. Implement lifecycle policies to transition old versions to cheaper
  storage tiers or expire them.

## 7. Security Considerations

- **Authentication:** CI/CD pipelines publishing artifacts **MUST** use secure authentication methods (e.
  g., OIDC with GitHub Packages/GHCR, short-lived tokens). Avoid long-lived PATs.
- **Vulnerability Scanning:** All code-based artifacts (libraries, container images) **MUST** be scanned
  for known vulnerabilities before publishing (see `cicd-001-baseline-workflow-guidance.md`).
- **Provenance:** Strive to generate and store build provenance information (e.g., using SLSA framework
  principles) to ensure artifact integrity and verify their origin. GitHub Actions has built-in support for
  attestation generation.
- **Access Control:** Ensure that only authorized CI/CD processes can publish artifacts to specific scopes
  or repositories within the artifact registry. Use GitHub Team permissions for GitHub Packages/GHCR.

## 8. Implementation in CI/CD

- Gem BB (DevOps Automation Specialist) is responsible for implementing artifact publishing and versioning
  steps within the reusable CI/CD workflows, adhering to this standard.
- Workflows **MUST** integrate tools for Conventional Commits parsing and SemVer bumping.
- Publishing steps **MUST** only run on protected branches (`main`, `release/*`) after all quality and
  security checks have passed.

## 9. Review and Updates

This standard will be reviewed periodically by the DevOps Team, Architects, and Lead Developers to adapt
to new tools, technologies, and project needs.

## **Chapter 17: General Debugging and Optimization**

* **Objective:** Provide the team with the tools and reflexes to identify and resolve bugs and performance bottlenecks. A non-performant game feels broken, and a hard-to-find bug can halt development for days. This chapter covers the "first-responder" tools to keep the project healthy.
* **Detailed Plan:**
  * **17.1. Mastering the Godot Debugger**
    * **Placing breakpoints:** This is the most fundamental debugging skill. In the script editor, click the gutter to the left of a line number to place a "breakpoint" (a red dot). When the game runs and reaches this line, it will *freeze* execution. This is your chance to inspect the "crime scene."
    * **Conditional Breakpoints:** Right-click a breakpoint to add a condition, like health \< 0 or item\_name \== "GodotSword". The game will *only* pause if the condition is true, saving you from having to step through a loop 1,000 times.
    * **Inspecting variables in real-time:** When paused at a breakpoint, the "Debugger" dock at the bottom of the editor comes alive. The "Inspector" tab shows you all local variables, member variables (self), and global variables. This is where you find the null reference (e.g., player\_target is null) or the incorrect value (e.g., damage is 0.0 instead of 10.0). You can also hover over variables directly in the script editor.
    * **The Call Stack:** This is the "breadcrumb trail" of how you got here. It shows the chain of function calls. For example, it might show that \_physics\_process called handle\_attack, which called apply\_damage, which is where your breakpoint is. This is critical for finding the *origin* of a bug (the "why"), not just where it finally crashed (the "what").
    * **Step Over, Step Into, Step Out:** Once paused, you have new controls.
      * **Step Over (F10):** Executes the current line and pauses on the *next line*. Use this to "step" through a function line by line.
      * **Step Into (F11):** If the current line is a function call (e.g., calculate\_path()), "Step Into" will dive *inside* that function and pause at its first line.
      * **Step Out (Shift+F11):** Finishes the current function and pauses in the function that *called* it.
  * **17.2. The Profiler: Your Best Friend for Performance**
    * **Identify functions that take the most computation time:** Open the "Debugger" dock and go to the "Profiler" tab. Click "Start" and let your game run during a demanding segment. The profiler measures *exactly* how many microseconds (ms) each function in your game is taking.
    * **"Self Time" vs "Total Time":** You can sort the function list by "Self Time" (how long a function *itself* took) or "Total Time" (how long it and all the *other functions it called* took). A high "Self Time" on a function in \_process is a "hotspot." This is your primary target for optimization. This is how you discover that your find\_nearest\_enemy() function is taking 90% of your frame's budget.
    * **Analyze memory usage:** The "Memory" tab in the profiler shows not just *how much* memory your game is using, but a count of every resource and node type. This is how you find "memory leaks." If you see the number of Bullet nodes constantly increasing (e.g., 50, 500, 5000...) and never decreasing, you know you are instancing them but forgetting to queue\_free() them when they go off-screen.
  * **17.3. The Remote Scene Monitor**
    * **Inspect and modify the scene tree of a running game:** While your game is running (even on a phone or in a separate window), click the "Remote" tab in the "Scene" dock. This shows you the *live* scene tree of the running game.
    * **Why it's powerful:** You can click on any node (e.g., select the Player) and see all its properties *live* in the "Inspector." You can watch its velocity vector change in real-time, or see if its is\_on\_floor property is "true" or "false."
    * **Live Modification:** You can even *change* these values. Is a bug hard to reproduce? Just "Remote" inspect the player, find their health property, and type 1 into it to test your "low health" visual effects or a "player died" bug. This is invaluable for debugging UI, physics states, and animations.
  * **17.4. Common Optimization Checklist**
    * **Culling:**
      * **Frustum Culling:** This is on by default. It stops the GPU from rendering objects that are *outside* the camera's view frustum (viewing cone).
      * **Occlusion Culling:** This is more advanced. It stops rendering objects that are *hidden behind other objects* (e.g., an enemy behind a building). This is disabled by default because it has a small CPU cost, but in complex, dense 3D scenes (like a city), it provides a massive GPU performance boost.
    * **Reduce Draw Calls:** A "draw call" is a command from the CPU to the GPU to "draw this mesh." Every object (a character, a tree, a rock) is one draw call. Thousands of draw calls will bottleneck the CPU.
      * **MultiMeshInstance:** This is the *key* to solving this. It allows you to render thousands (or millions) of *identical* meshes (like a forest of trees, an asteroid field, or a crowd) in a *single* draw call.
      * **Atlases (2D):** For 2D, instead of using 100 separate 32x32px image files, use one 1024x1024 "Sprite Atlas" (a single image sheet). This allows the renderer to batch all those sprites into one draw call, dramatically improving 2D performance.
    * **Optimize Physics:**
      * **Use the right node:** Don't use a RigidBody3D for a wall. StaticBody3D is for all non-moving collision (floors, walls, ramps). The physics engine *heavily* optimizes for these. RigidBody3D is for complex, physics-driven objects (crates, barrels) and is much more expensive. CharacterBody3D is for player/enemy movement using move\_and\_slide().
      * **Physics Layers & Masks:** This is a huge optimization. By default, everything can collide with everything. Use layers to be specific. Put your "Bullets" on layer 3 and make them *only* check for collisions on the "World" (layer 1\) and "Enemies" (layer 2\) masks. This stops bullets from checking for collisions with other bullets or trigger zones, saving CPU time.
    * **Optimize Code:**
      * **Signals vs. \_process:** (Reiterated from Anti-Patterns) Do *not* have 100 enemies in \_process check if player.global\_position.distance\_to(self.global\_position) \< 10: attack(). This is 100 distance checks every single frame. The "Godot-idiomatic" way is to give each enemy an Area3D node. The physics engine will emit the body\_entered signal *once* when the player gets close, which *triggers* the attack logic. This is event-driven and infinitely more performant.
      * **Cache Node References:** Do not call get\_node("AnimationPlayer").play("run") inside a function that runs every frame. The get\_node() call is a string-based lookup that has to search the scene tree. It's slow. Instead, call it *once* in \_ready() and store it in a variable: var anim\_player. Then, just use anim\_player.play("run"). This is thousands of times faster.

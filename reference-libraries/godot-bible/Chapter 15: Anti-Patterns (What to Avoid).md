## **Chapter 15: Anti-Patterns (What to Avoid)**

* **Objective:** Identify, name, and explain common design flaws to be able to spot and correct them quickly during code reviews. Understanding *what not* to do is as important as understanding what to do, as it helps build maintainable, scalable, and performant projects. These patterns are "traps" that seem convenient at first but create significant problems as the project grows.
* **Detailed Plan:**
  * **15.1. The "Mega-Script"**
    * **Description:** A single script, often attached to the "Player" node (or a "Level" node), that has grown to thousands of lines. It attempts to be the "brain" of everything. It handles input, physics, state management (idle, walking, jumping, stunned, attacking), animation, inventory logic, sound effects, collision responses, and sometimes even direct UI updates like modifying label text. The \_process and \_physics\_process functions become massive, unreadable match statements or a long series of if-elif-else checks, and the script has dozens of member variables just to track its many sub-states.
    * **Consequences:** This script becomes impossible to read, debug, or maintain. It has zero "cohesion" and infinite "coupling." A small change in the inventory logic can accidentally break the physics state. When a bug occurs (e.g., "player can jump while stunned"), the logic could be hidden in one of five different functions, all interacting in unpredictable ways. The cognitive load is too high; no single person can hold the entire script's logic in their head. It also makes collaboration impossible; an artist trying to tweak animation-related function calls will constantly get merge conflicts with a programmer trying to fix physics bugs, as they are both editing the same enormous file. It fundamentally violates the Single Responsibility Principle and completely ignores Godot's node-based, compositional philosophy.
    * **Solution:** Use composition and node-based design. Break the logic into discrete components, each with its own node and script, following the "Scene as a tree of components" paradigm. The "Player" is not *one script*; it's a *scene* (a tree) of specialized nodes working together.
      * **Refactored Player Scene:**
        * Player (a CharacterBody3D) handles *only* move\_and\_slide() and core physics. Its script should be small, primarily exposing properties like max\_speed and functions like apply\_knockback(vector).
        * InputController (a child Node) handles *only* processing Input events. It knows nothing about physics or animation. It just emits signals like jump\_pressed or move\_vector\_changed(vector).
        * AnimationHandler (a child Node with an AnimationTree) listens for signals from the InputController and Player (e.g., is\_on\_floor\_changed) to update animation parameters. It is the *only* thing that knows or cares about the AnimationTree.
        * Inventory (a child Node) manages a list of items and has its own clear API (add\_item, remove\_item). It emits signals like inventory\_updated.
        * PlayerAudio (a child Node with several AudioStreamPlayers) listens for signals (jump\_pressed, landed, took\_damage) from all the other components and plays the appropriate sounds.
  * **15.2. The get\_node() Hell**
    * **Description:** Using hard-coded string paths to access other nodes, such as get\_node("../../../UI/HUD/ScoreLabel").text \= str(score) from a script deep in the Level/Player scene. This also includes fragile get\_parent().get\_parent() chains. This often happens because it's the "quickest" way to make two nodes talk in the editor.
    * **Consequences:** This is extremely fragile. If any node in that path is renamed, moved, or reparented during scene reorganization, the path breaks and the game crashes with a "Node not found" error. This makes refactoring a nightmare. Imagine you want to put the 'HUD' inside a 'UI\_Container' for better scaling. You make this simple change, and suddenly 15 different scripts crash because their paths are now wrong. You can't reuse the Player in a different level that has a different UI structure, as the hard-coded path will fail. It tightly couples the script to the scene's exact implementation details.
    * **Solutions:**
      * **@export vars:** For nodes within the *same* scene, this is the best method. Use an exported variable: @export var score\_label: Label. You can then drag-and-drop the ScoreLabel node onto this variable in the Inspector. This creates a direct reference that is 100% robust to renames and path changes. The engine maintains the reference for you.
      * **%UniqueName:** If you must use get\_node, mark the target node as a "Unique Name" in the scene (the '%' icon) and access it via %ScoreLabel. This is robust as long as the node remains in the *same scene*. It is still less ideal than @export as it's a string-based lookup at runtime, but it's much better than a path.
      * **Signals:** For cross-scene communication (e.g., a Player in Level.tscn needs to talk to a HUD in Main.tscn), signals are the only robust solution. Do not let the Player "reach out" to the UI. The UI should listen for the Player. (See 15.4).
  * **15.3. The Abusive Singleton**
    * **Description:** Using Godot's "Autoload" feature (singletons) to store all global game logic and state. This often manifests as a GameManager.gd or Globals.gd that holds the player's health, current score, level name, is\_paused, current\_quest\_id, a direct reference to the player node itself (GameManager.player), and lists of all enemies (GameManager.all\_enemies).
    * **Consequences:** This is just a "global variable bucket" and creates a "God Object." It leads to hidden, untraceable dependencies. *Any* script can access and modify GameManager.player\_health at *any* time. When you have a bug where health changes unexpectedly, the cause could be in *any script in the entire project*. It also makes individual scenes untestable; you can't run a test scene for an Enemy because its \_ready function might try to access GameManager.player, which is null when run in isolation. This friction actively discourages testing and leads to more bugs.
    * **Solution:** Use Autoloads *only* for things that are truly global, singular *services*, not data containers. Distinguish *services* (things that *do* something) from *data* (things that *are* something).
      * **Good Singletons (Services):** SoundManager (with functions like play\_music(track)), SceneTransitioner (with fade\_to\_scene(path)), GlobalSettings (managing volume, keybinds).
      * **Bad Singletons (Data that belongs elsewhere):**
        * player\_health belongs *on the Player node*.
        * score might be managed by a ScoreManager node *in the main level scene*, not globally.
        * A reference to the player is unnecessary; nodes that need the player can use groups (get\_tree().get\_first\_node\_in\_group("player")) or signals. The player's data belongs *on the player*.
  * **15.4. The Direct Reference Obsession**
    * **Description:** Ignoring the signal system to couple objects directly. This is the *architectural* version of get\_node() Hell, often born from a "get-and-call" mindset. It's the "need" for the Player script to have a direct variable var ui \= get\_node("%HUD") so it can call ui.update\_health(health). The Player "knows" about the HUD and its specific function names.
    * **Consequences:** This creates "spaghetti code" and tight coupling. The Player is now dependent on the HUD. You can't instance the Player in a test scene without a HUD present, or it will crash. What if you want to add a *second* health bar for a spectator, or a "camera shake" effect when the player takes damage? You would have to modify the Player script to add ui\_spectator.update\_health(health) and camera.shake(). The Player script becomes bloated with responsibilities that aren't its own. Its job is to *be* a player, not to manage the camera and UI.
    * **Solution:** Invert the dependency with signals (the Observer Pattern). The Player should know *nothing* about the UI or other systems. It should simply announce its state changes to the world.
      * Player.gd: signal health\_changed(current\_health, max\_health) ... func take\_damage(amount): health \-= amount; health\_changed.emit(health, max\_health)
      * HUD.gd: In \_ready(), it finds the player (e.g., get\_tree().get\_first\_node\_in\_group("player")) and connects to its signal: player.health\_changed.connect(\_on\_player\_health\_changed).
      * CameraShake.gd: In \_ready(), it also connects: player.health\_changed.connect(\_on\_player\_took\_damage).
      * Now, the Player *emits* signals, and the HUD and CameraShake *listen*. They are loosely coupled. The Player doesn't know or care who is listening. You can add 10 more systems that react to health\_changed without *ever* touching Player.gd again.
  * **15.5. The UI That Thinks**
    * **Description:** Storing the "source of truth" for game state *in* the UI nodes. For example, a HealthBar.gd script has a variable var current\_health \= 100\. When an enemy hits the player, it calls get\_node("%HealthBar").current\_health \-= 10\. Another example: an AmmoCountLabel.gd script has var ammo \= 30, and the Weapon script calls get\_node("%AmmoCountLabel").ammo \-= 1 on fire.
    * **Consequences:** This is a critical design flaw. The UI is now responsible for game logic. What happens if current\_health reaches 0? Does the HealthBar script now call get\_tree().reload\_current\_scene()? What if the player picks up an ammo pack? The AmmoPack script now *also* needs a reference to AmmoCountLabel. The game's state is fragmented, spread across UI elements, and stored in the wrong place, making it impossible to manage, save, or load. If you want to pause the game, do you have to find the "PauseButton" and ask it to pause? This is completely backward.
    * **Solution:** The UI should be "dumb." It *only* visualizes state that is held elsewhere. It is a *view* of the *model*, not the model itself. The Player script is the "source of truth" for health. The Weapon script is the "source of truth" for ammo. The HealthBar script should have *no* state variables, only a function to *receive* state, which is connected to the Player's health\_changed signal.
      * HealthBar.gd: func \_on\_player\_health\_changed(current\_health, max\_health): value \= float(current\_health) / max\_health
      * AmmoCountLabel.gd: func \_on\_weapon\_ammo\_changed(new\_ammo): text \= str(new\_ammo)
  * **15.6. (Multiplayer) The All-Powerful Client**
    * **Description:** Trusting any information sent from a client. This includes RPCs (Remote Procedure Calls) like server.set\_my\_health(100), server.i\_got\_a\_kill(), server.my\_position\_is(Vector3(10, 5, 2)), or server.i\_bought\_item("legendary\_sword"). This also includes subtle things, like a client sending i\_opened\_treasure\_chest(chest\_id).
    * **Consequences:** This is a massive, fundamental security and cheating vulnerability. A malicious user can modify their client to send any RPC they want, giving themselves infinite health, teleporting, or instantly winning the game. You are not building a game; you are building a suggestion box that hackers can ignore. Even in a co-op game, this leads to desynchronization. If a client opens a chest it wasn't "supposed" to (e.g., it didn't have the key, but its local check was buggy), the server now has a corrupt state.
    * **Solution:** Server-Side Authority. The server is the *only* source of truth. The client *never* dictates state; it *only* sends its *input* or *intent*.
      * **Bad:** client \-\> server.rpc("shoot\_bullet", bullet\_position, bullet\_direction)
      * **Good:** client \-\> server.rpc\_id(1, "player\_pressed\_shoot", look\_direction)
      * The server then receives this "intent." It checks its own internal, authoritative game state: Is the player alive? Do they have ammo? Is their weapon off cooldown? If all checks pass, the *server* then spawns the bullet, performs the physics trace, and *tells all clients* (including the one who sent the request) via a separate RPC that a bullet was spawned and what it hit.
      * **Client-Side Prediction:** To make this feel responsive, the client can "predict" the outcome. When the user clicks, the client *locally* plays a sound and spawns a *visual-only* effect. This is the prediction. When the server's *authoritative* message arrives, the client replaces its visual-only effect with the "real" networked object. If the server denied the request (e.g., no ammo), the client simply cancels its prediction.
  * **15.7. (Performance) The \_process Calculation**
    * **Description:** Putting heavy, complex logic inside the \_process(delta) or \_physics\_process(delta) functions. This includes things like A\* pathfinding, iterating over thousands of array items (e.g., for enemy in get\_tree().get\_nodes\_in\_group("enemies"): if (enemy.global\_position.distance\_to(global\_position) \< 10): ...), complex AI decision-making, or loading resources from disk (load("res://...")).
    * **Consequences:** This code runs 60 times per second (or more), even if nothing has changed. This causes a constant, high CPU load. A single frame that takes too long (a "frame spike") is perceived by the player as a "hitch" or "stutter," which is far more jarring than a slightly lower but consistent frame rate. This is "death by a thousand cuts"; one script doing this is fine, but fifty scripts doing it will kill your performance, especially on mobile/web.
    * **Solutions:**
      * **Event-Driven Logic:** Run the heavy logic *only when it's needed*. Don't check distance to all enemies every frame; use an Area3D node and its body\_entered signal to detect when an enemy is *actually* nearby. The C++ physics engine is infinitely faster at this than your GDScript loop. Don't run pathfinding every frame; run it *once* when the enemy's state changes to "chasing," and then simply follow the resulting path.
      * **Timers:** If you have an AI that needs to "think" (e.g., re-evaluate its strategy), don't make it think every frame. Use a Timer node and connect its timeout signal to the \_on\_think function. Set it to run every 0.5 or 1.0 seconds. This batches the thinking process and spreads the CPU load.
      * **Threading:** For truly massive, non-interruptible tasks (like procedural mesh generation or complex physics calculations), use a Thread to run the calculation in the background. **Warning:** Godot's API is not thread-safe. The thread *cannot* access nodes (get\_node) or change properties. It must perform pure calculations on data and then use call\_deferred or emit a signal to pass the *result* back to the main thread, which is the only one allowed to touch the scene tree.
      * **Time-Slicing:** If you must process a huge list, do it in chunks. Process 100 items from the list per frame, spreading the load over many frames. Use a counter variable to track your index in the list and await get\_tree().process\_frame to yield.

# KB: Technical Documentation Standards

**Version:** 1.0.0
**Last Updated:** 2025-06-13
**Status:** Approved
**Knowledge Guardian:** @Isaac (GCT-PRG-SARCH-001)

## 1. Purpose and Scope

This knowledge base article defines the standards and best practices for creating and maintaining all **technical documentation** within GenCr-ft Studio. It ensures clarity, consistency, and traceability across codebases, software systems, APIs, and architectural decisions, supporting both human developers and AI Gems.

The scope of this document includes:

- Types of technical documentation and their purpose (e.g., TDDs, API specifications, system overviews).
- Standards for documenting code, configurations, and infrastructure.
- Guidelines for tools, platforms, and formats used in technical documentation.
- Processes for technical documentation review, approval, and maintenance.
- Integration with the software development lifecycle (SDLC) and CI/CD pipelines.

This document serves as a foundational reference for all Programming Teams, Architects, DevOps Teams, and any Gem involved in the technical aspects of GenCr-ft's development.

## 2. Key Information / Concepts / Procedures

### 2.1. Types of Technical Documentation

GenCr-ft utilizes various forms of technical documentation, each serving a specific purpose:

- **Technical Design Documents (TDDs):** In-depth documents detailing the design, implementation, and rationale behind specific software components, features, or systems.
  - SSoT: `gcp-aethel-architecture/tdd/` (for game-specific TDDs) or `gcs-plt-architecture/tdd/` (for platform-level TDDs).
  - Template: `technical-design-document-template.md`.
- **API Specifications:** Formal descriptions of internal and external APIs, detailing endpoints, request/response formats, authentication, and error codes.
  - SSoT: `gcp-aethel-architecture/api-spec/` or within project repositories for specific service APIs.
  - Template: `mcp-server-api-spec-template.md`.
- **System Overviews / Architecture Diagrams:** High-level descriptions of system components, their interactions, and data flows, often accompanied by visual diagrams (e.g., C4 model).
  - SSoT: `https://github.com/GenCr-ft/gcp-aethel-architecture/blob/main/README.md`.
- **Code Documentation:** Comments within code, docstrings, and autogenerated documentation providing context for individual functions, classes, modules, and libraries.
- **Infrastructure as Code (IaC) Documentation:** Explanation of Terraform/OpenTofu modules, cloud resource configurations, and deployment environments.
  - SSoT: `https://github.com/GenCr-ft/gcs-core-governance/blob/main/iac/README.md`.
- **Configuration Documentation:** Details of application and environment configurations, environment variables, and feature flags.
- **Troubleshooting Guides / Runbooks:** Practical guides for diagnosing and resolving common operational issues (often managed by DevOps and SRE teams).

### 2.2. Technical Documentation Standards

All technical documentation **must** adhere to the following overarching standards:

- **Clarity and Precision:** Use unambiguous language. Avoid jargon where possible, or define it in the `glossary.md`.
- **Accuracy:** Documents must accurately reflect the current state of the system, code, or architecture. Outdated documentation is misleading and dangerous.
- **Consistency:** Maintain uniform terminology, formatting, and structure across all related documents and within the codebase.
- **Completeness:** Provide sufficient detail for the intended audience to understand and utilize the information effectively.
- **AI Parsability:** Structure documents semantically (using headings, lists, tables) and provide rich YAML frontmatter to facilitate interpretation by AI Gems (`GASAI_A3_SSOT_005`). Refer to `GOV-GUIDE-007.knowledge-management-and-contribution-guide.md`.
- **Traceability (Protocol S7):** Link documents to relevant requirements, design decisions, issues, and code changes.
- **Security by Design (Protocol S8):** Avoid including sensitive information directly in documents. Classify documents using `confidentialityLevel` and enforce access controls.

### 2.3. Tools and Platforms

GenCr-ft employs a set of standardized tools and platforms for technical documentation:

- **Markdown:** Primary format for all prose-based technical documentation within Git repositories.
- **YAML Frontmatter:** Mandatory for metadata in all Markdown SSoT documents.
- **Git & GitHub:** Version control system for all technical documentation, enabling collaborative review via Pull Requests.
- **Mermaid:** For generating diagrams (flowcharts, sequence diagrams, state diagrams) directly within Markdown documents for versionability.
- **Doxygen/Sphinx/Jsdoc (or similar):** For generating API documentation directly from code comments/docstrings for various programming languages.
- **Confluence/Wiki (complementary):** May be used for dynamic content, collaborative brainstorming, or less formal technical notes, but the SSoT principles always refer to Git-versioned Markdown.

### 2.4. Code Documentation Best Practices

Effective code documentation is crucial for maintainability and AI comprehension.

- **Inline Comments:** Use comments to explain the "why" behind complex logic, design decisions, workarounds, or critical assumptions (refer to `GOV-GUIDE-007.knowledge-management-and-contribution-guide.md` Section 6).
- **Docstrings / Block Comments:** For functions, classes, and modules, provide a clear, concise docstring or block comment explaining:
  - Purpose/functionality.
  - Parameters (type, description).
  - Return values.
  - Pre-conditions/Post-conditions.
  - Examples of usage.
- **Consistent Style:** Adhere to the language's conventional style guides (e.g., PEP 8 for Python) and internal project-specific style guides.
- **Maintain with Code:** Documentation (comments, docstrings) **must** be updated concurrently with code changes to prevent drift. Automated linters and review processes enforce this.

### 2.5. Technical Documentation Review and Approval

All technical documentation follows the Studio's standard document generation and approval workflows.

- **Document Generation Process (DEVPROC_001):** New technical documents or major revisions follow the structured `DEVPROC_001` workflow, involving drafting, iterative review, and automated quality checks.
- **Feedback & Approval (Protocol S1):** All technical documentation is submitted via Pull Requests and requires approval from designated reviewers and Knowledge Guardians (`Isaac` and relevant leads for programming domains).
- **Reviewers:** Include relevant technical leads (`Julien`, `Rémi`), domain experts, architects, and security specialists (`Cerberus`).
- **Automated Checks:** CI/CD pipelines automatically validate Markdown formatting, YAML frontmatter, and link integrity for technical documentation.

### 2.6. Integration with Development Lifecycle

Technical documentation is an integral part of the SDLC, not an afterthought.

- **Requirements Phase:** TDDs and system overviews start from requirements (`gcp-aethel-docs-req`).
- **Design Phase:** Architecture decisions (`gcp-aethel-architecture/adrs/`) and TDDs are created.
- **Implementation Phase:** Code documentation is written concurrently with code.
- **Testing Phase:** QA teams leverage TDDs and API specs for test case design.
- **Deployment Phase:** Runbooks and deployment guides are finalized.
- **Agile Integration (Protocol S15):** Documentation updates are treated as tasks/stories within sprints. "Definition of Done" for features includes updated documentation.
